# Token Governance

Tokens are a shared resource. To maintain the integrity of the Daily Mail Design System, all token additions, modifications, and deprecations must follow the process below.

---

## 1. Adding a New Token

Before requesting a new token, check whether an existing token can serve the purpose. Unnecessary tokens increase system complexity.

- **Proposal:** Submit a brief outline to the Design System Lead describing the token's intended use, the UI context, and why existing tokens are insufficient.
- **Review:** The design system team will assess whether the token warrants a Semantic, Categorical, or Miscellaneous classification, and validate it against WCAG 2.1 AA in both Light and Dark modes.
- **Approval & Integration:** Once approved, the token will be added to the Master Figma library, JSON source, and documentation simultaneously.

---

## 2. Modifying an Existing Token

Token values should only be updated when there is a clear accessibility, brand, or product need.

- All modifications require sign-off from the Design System Lead and Head of Product Design.
- Impact across UK, US, and AU platforms must be assessed before any change is merged.

---

## 3. Deprecating a Token

A deprecated token will enter a grace period before removal, during which it remains functional but is flagged in documentation.

Teams using the deprecated token will be notified via the Product Design team channel to migrate to the recommended replacement.

---

## 4. Promoting a Miscellaneous Token

If a Miscellaneous token begins appearing across multiple products or contexts, it should be reviewed for promotion to a Semantic or Categorical token. Raise this with the Design System Lead for assessment.

---

> 💡 For queries or to raise a proposal, contact the Design System Lead or post in the Daily Mail Product Design team channel.
